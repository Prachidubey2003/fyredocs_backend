# Secrets — inventory, lifecycle, rotation

This document is the operator's runbook for every secret the Fyredocs
backend depends on: where it lives, who can read it, how to rotate it,
and what to do when one leaks. Keep it short and current — a stale
secrets doc is worse than none at all.

The companion policy is [`../../../SECURITY.md`](../../../SECURITY.md).

## Inventory

Each secret has an **owner** (the engineer or team responsible) and a
**rotation cadence** (the upper bound between scheduled rotations,
regardless of leaks).

| Secret | Used by | Stored in | Owner | Routine rotation | Emergency rotation procedure |
|---|---|---|---|---|---|
| `DATABASE_URL` (Postgres password embedded) | every service | `.env` on each host / K8s `Secret` later | DBA | **90 days** | §1 |
| `POSTGRES_PASSWORD` | bootstrap of self-hosted Postgres | `.env` on the DB host | DBA | **90 days** | §1 |
| `REDIS_PASSWORD` | every service | `.env` / K8s `Secret` | Platform | **90 days** | §2 |
| `JWT_HS256_SECRET` | `auth-service` (signs), every gateway/service (verifies) | `.jwt_secret` generated by [`deployment/deploy.sh`](../../../deployment/deploy.sh), distributed via env | Security | **30 days** (short — signs sessions) | §3 |
| `NATS` auth (when enabled) | every service | `.env` / NATS operator JWT file | Platform | **90 days** | §4 |
| Restic backup password + bucket creds (`RESTIC_REPOSITORY`, `B2_ACCOUNT_*`) | storage host | `/etc/fyredocs/restic.env` (mode `0400` root) | SRE | **365 days** (long — losing it doesn't break the running system) | §5 |
| GHCR write token (for `.github/workflows/docker.yml`) | CI | GitHub Actions secret `GITHUB_TOKEN` (auto-issued) | n/a | per-job (auto) | rotate by re-running the workflow |
| Dependabot / CodeQL tokens | CI | GitHub-managed | n/a | n/a | n/a |
| TLS certificates (eventually for the gateway) | api-gateway | `/etc/fyredocs/tls/` | Platform | **90 days** (ACME auto-renew) | reissue via ACME |

**Never** stored in this repo:
- Production `.env` files. The repo only contains [`.env.example`](../../../.env.example).
- `.jwt_secret`. Gitignored by [`.gitignore`](../../../.gitignore).
- Restic credentials.

If you find a real secret in the git history, treat it as leaked and follow [§7 — incident response](#7-incident-response).

## Generating new values

| Secret | Generate |
|---|---|
| `JWT_HS256_SECRET` | `openssl rand -hex 32` — 32-byte / 64-hex-char random. Min length 32 bytes. |
| `POSTGRES_PASSWORD` / `REDIS_PASSWORD` | `openssl rand -base64 32 \| tr -d '/+= '` — alphanumeric, 30+ chars. |
| Restic repo password | `openssl rand -base64 48` — long; you only type it for restore. |
| NATS operator/account JWT | `nsc add ...` per the NATS operator docs. |

All generators are entropy-from-`/dev/urandom`; do not type them yourself.

## Distribution

- **Dev**: copy [`.env.example`](../../../.env.example) → `.env`, fill values locally. Gitignored.
- **Staging/Prod (current)**: `deploy.sh` reads from `.env` on the host; secrets stay on disk with `0600` ownership.
- **Future (K8s)**: K8s `Secret` resources synced from Vault via `external-secrets`. Per-namespace RBAC limits which service accounts can mount which Secrets.

CI (GitHub Actions) never touches production secrets. It only needs the auto-issued `GITHUB_TOKEN` and, for SBOM/security uploads, the built-in `security-events: write` permission.

## Rotation procedures

Each procedure assumes you have shell access to the host or `kubectl` to the cluster. **All rotations are written to be zero-downtime by default.** If you need downtime, take the maintenance-window path explicitly.

### §1 — Postgres password (`DATABASE_URL`, `POSTGRES_PASSWORD`)

1. Generate the new password (see above).
2. Apply it to the database while keeping the old password valid via `scram-sha-256` dual-credential window:
   ```sql
   ALTER ROLE fyredocs_admin PASSWORD '<new>' VALID UNTIL 'infinity';
   ```
3. Update `DATABASE_URL` on each service host (or in `external-secrets`) and `systemctl restart <service>` / `kubectl rollout restart deployment/<service>` one at a time. Watch `/healthz` and Postgres `pg_stat_activity` between rollouts.
4. After every service has reconnected, scan `pg_stat_activity` for any connections still using the old credential (they would have failed and been replaced). Once clean:
   ```sql
   -- Optional: revoke the old credential by setting a no-op password it can't match.
   -- With single-password Postgres, the rotate-in-place above already invalidated the old one.
   ```
5. Update the password in your password manager. Log the rotation in the security log.

**Rollback**: if a service can't reconnect, revert `DATABASE_URL` to the previous value (kept in the password manager for 24h) and `ALTER ROLE ... PASSWORD '<old>'` until the cause is found.

### §2 — Redis password

1. Generate the new password.
2. Configure Redis for **dual-password** (Redis 7 ACL): `CONFIG SET requirepass <new>` and update the ACL for the `default` user — but better practice is **per-service ACL users**.
   ```
   ACL SETUSER fyredocs >oldpassword >newpassword on +@all
   ```
3. Rollout services one at a time.
4. Once all services use the new password, drop the old one: `ACL SETUSER fyredocs <oldpassword off`.
5. Update the password manager.

**Rollback**: ACL keeps both passwords active during the window; revert by re-enabling the old password.

### §3 — JWT HS256 signing key (`JWT_HS256_SECRET`)

Dual-key verification is implemented — see [`api-gateway/internal/authverify/verifier.go`](../../../api-gateway/internal/authverify/verifier.go), [`auth-service/internal/authverify/verifier.go`](../../../auth-service/internal/authverify/verifier.go), [`job-service/internal/authverify/verifier.go`](../../../job-service/internal/authverify/verifier.go), and [`auth-service/internal/token/issuer.go`](../../../auth-service/internal/token/issuer.go) `VerifyRefreshToken`. Plan for a window equal to `JWT_ACCESS_TTL` (default 8h) before removing the previous key.

1. Generate `<new>` (see "Generating new values").
2. Configure dual-key verification: set `JWT_HS256_SECRET=<new>` and `JWT_HS256_SECRET_PREVIOUS=<old>` on every service that touches JWTs (api-gateway, auth-service, job-service). The verifier accepts tokens signed with either key; new tokens are signed only with `<new>`.
3. Roll out auth-service first (so new tokens get the new signature), then every other service in any order.
4. Wait `JWT_ACCESS_TTL` so every issued token using the old key has expired or been refreshed.
5. Remove `JWT_HS256_SECRET_PREVIOUS` from every service. Roll out.
6. Update the password manager + the security log.

**During the window**: refresh tokens still signed with the old key continue to mint new access tokens with the new signing key (via `Issuer.VerifyRefreshToken`'s dual-key fallback). No user is forcibly logged out.

**Rollback**: leave `JWT_HS256_SECRET_PREVIOUS` deployed until the rotation is fully bedded in. Worst-case revert by swapping the env vars and rolling forward.

**Coverage**: the `Verifier` retries with `PreviousHMACSecret` only on `jwt.ErrTokenSignatureInvalid` — expired tokens, malformed tokens, denylisted JTIs, and tokens with the wrong issuer/audience still fail without fallback. RSA keys are out of scope for this rotation flow.

### §4 — NATS credentials

1. Issue a new account JWT via `nsc`.
2. Push the JWT to the NATS server (or the deployed credentials file).
3. Restart services with the new creds; the NATS server briefly accepts both during overlap if configured with operator JWT rotation.
4. Revoke the old account.

### §5 — Restic repository password + bucket creds

These never reach production services; they only sit on the storage host(s). Rotation is mechanical:

1. `restic key add` — adds a new key alongside the old one.
2. Update `/etc/fyredocs/restic.env` with the new bucket creds (if rotating those) and the new restic password file.
3. Re-run [`deployment/storage/restic-backup.sh`](../../../deployment/storage/restic-backup.sh) once manually to confirm it still works.
4. `restic key remove <old-id>`.

### §6 — GHCR / GitHub Actions tokens

These are auto-issued per workflow run by GitHub (`GITHUB_TOKEN`). You don't rotate them; you rotate by re-running the workflow.

If you've added a third-party PAT to repo secrets (e.g., for a non-GHCR registry), rotate it from the GitHub UI: `Settings → Secrets and variables → Actions`.

## §7 — Incident response (a secret has leaked)

Treat any of the following as a confirmed leak:

- gitleaks fires on a non-allowlisted match.
- A team member confirms a secret was pasted in Slack / email / a chat tool.
- A scanner (TruffleHog, GitHub secret scanning) opens a security advisory.
- An ex-employee retains a copy of `.env`.

Within **2 hours**, the on-call:

1. **Rotate the secret immediately** following §1–§5. Do not wait for the next routine cycle.
2. **Invalidate any sessions/tokens issued under the leaked secret** (for `JWT_HS256_SECRET`: this means cycling without the dual-key window; users get logged out).
3. **Open an incident ticket** with: scope, suspected timeline of exposure, affected systems, customer-data implications.
4. **Notify legal/compliance** if customer data is reachable through the leaked credential (HIPAA: 60 days; GDPR: 72 hours).
5. **Purge from history**: `git filter-repo` if leaked into a public repo; otherwise force-push is acceptable on a private repo *only after* every clone has been updated.
6. **Write a postmortem** within 7 days. Blameless format; root cause + mitigations land in the next sprint.

The on-call rotation, escalation contacts, and oncall paging policy live in `runbooks/oncall.md` (to be added in Phase 5 enterprise tier — until then, escalate to the email listed in [SECURITY.md](../../../SECURITY.md)).

## Per-document envelope encryption

For sensitive-tier tenants (per plan §4.4.6 / HIPAA BAA § 5.3), document bodies are envelope-encrypted: each document body is sealed with a fresh per-document DEK (32-byte AES-256 key) generated at write time, and that DEK is wrapped with the tenant's KEK before storage. The wrapped DEK lives next to the ciphertext in Postgres; the KEK lives in this service's secret store (Vault for prod, SOPS-decrypted env for dev).

The cryptographic primitive is in [`shared/keystore/`](../../../shared/keystore/envelope.go):

| Function | Purpose |
|---|---|
| `GenerateDEK() → []byte` | Fresh 32-byte AES-256 key from `crypto/rand`. |
| `WrapDEK(kek, dek, randSrc) → []byte` | Encrypts the DEK with the tenant KEK; returns a fixed-size 60-byte blob (`nonce ‖ ciphertext ‖ gcmTag`). |
| `UnwrapDEK(kek, wrapped) → []byte` | Reverses WrapDEK. Auth-tag failure surfaces as `ErrAuthFailed` — same error for "wrong KEK", "wrong ciphertext", or "tampered bytes" so failure modes can't be distinguished by side-channel. |
| `SealWithDEK(dek, plaintext, randSrc) → []byte` | Document-body encryption. Same nonce-prefix wire format as WrapDEK. |
| `OpenWithDEK(dek, sealed) → []byte` | Reverses SealWithDEK. |
| `RewrapDEK(oldKEK, newKEK, wrapped, randSrc) → []byte` | Key-rotation primitive — unwraps with the old KEK and re-wraps with the new one. Avoids re-encrypting (potentially large) document bodies on KEK rotation. |

All AES-256-GCM. Integer constants exported (`KeySize`, `NonceSize`, `GCMTagSize`, `WrappedDEKSize`). The library is pure — no IO, no global state, no key custody. Auth-service / editor-service hold the KEK in memory and call into this library; rotation cadence + Vault adapter wiring live with the keystore-infra work.

Pinned invariants (see [`shared/keystore/envelope_test.go`](../../../shared/keystore/envelope_test.go)):
- Two `WrapDEK` calls on the same `(KEK, DEK)` produce different ciphertext — the nonce is fresh per call, so a regression that reuses nonces would break AES-GCM's security and fail loudly.
- Tampered nonce, tampered ciphertext, and wrong KEK all surface as `ErrAuthFailed` (uniform failure mode).
- Wrong-size keys, truncated wrapped DEKs, and too-short sealed bodies fail-fast with their own typed sentinels.
- `RewrapDEK` produces a new wrapping that opens with the new KEK *and refuses to open with the old one* — the rotation is real, not a no-op.
